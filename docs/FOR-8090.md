## TL;DR

I studied the Software Factory's architecture — **Requirements → Blueprints → Work Orders → Tests →
Knowledge Graph** — and noticed it runs *forward* beautifully but has no **return path**. The hard-won
context an agent or engineer generates while delivering a Work Order evaporates the moment the session
closes. The next session starts cold and re-learns what the last one already knew.

So I designed and shipped **CAPSULE**: the missing **capture + feedback (reinforcement) layer** that turns
that one-way pipeline into a closed loop. It distills each finished session into a scored, versioned
**capsule**, stores it as portable memory, compounds capsules into **versioned enterprise skills**, and
feeds them back so every future session starts *warm*. The reward is measured in tokens saved — the loop
literally pays for itself.

I built it solo, end-to-end: product thesis, architecture, a working Next.js app, the local-model
distillation pipeline, the multi-developer promotion/CI/governance model, Docker packaging, and the docs you
are reading. **It is not a mock** — it captured its own build session and promoted that into the enterprise
registry. That self-closing loop is the proof it runs.

## The gap I found in the Software Factory

8090's thesis is that the leverage in software is **upstream** — requirements, architecture, planning — not
in generating more code faster. I agree, and CAPSULE extends that thesis one step further:

> If the leverage is upstream, then the **context** that produced good upstream decisions is the most
> valuable artifact in the building — and today it is the one thing the pipeline throws away.

Every component of the Factory is served by closing this:

| 8090 component | What CAPSULE adds |
|---|---|
| **Requirements** | Capsules and skills stay traceable to the requirement they served, so retrieval at session-open biases toward the active requirement's prior context. |
| **Blueprints** | Past decisions and gotchas surface *before* a new design is drafted — blueprints stop re-litigating solved questions. |
| **Work Orders** | When a Work Order completes, its session capsule is minted automatically; open Work Orders scope what gets recalled next. |
| **Tests** | Gotchas become regression guards. The A/B harness *is* an agentic-CI gate that re-checks prior capsules before a skill version publishes. |
| **Knowledge Graph** | CAPSULE renders as an overlay on the native graph, adding `capsule`, `skill@version`, and `memory` nodes plus the edges that close the loop. |

This is the same intuition behind 8090's **Assembly Lines** — "memorize and automate a pattern so you can
repeat it with increasing accuracy." CAPSULE is the mechanism that *learns* those patterns from real work
instead of requiring someone to hand-author them.

## What I built

A real **Next.js 15 / React 19 / TypeScript** app with a real server-side pipeline:

```
session  →  CAPTURE  →  DISTILL (local Ollama)  →  SCORE (LLM-judge)  →  GATE
         →  STORE (Backboard memory)  →  LOCAL REGISTRY (git)  →  EOD PROMOTE (CI-gated PR)
         →  ENTERPRISE REGISTRY  →  token savings feed the next session
```

- **Local-first distillation.** Capture runs on-device on `qwen2.5-coder:14b` — private, free, offline,
  no transcript egress. That is what makes capturing *every* session affordable, which is what makes the
  reward signal real instead of aspirational. Cerebras is an optional cloud boost; a heuristic is the
  last-resort backfill. The whole app runs with **zero API keys** via graceful degradation.
- **Portable memory.** Capsules persist to Backboard keyed by **entity (tenant), not model** — switch any
  of 17,000+ models and the memory re-fits, never re-derives. Store-only writes (`send_to_llm=false`) mean
  capture can never exfiltrate through an LLM round-trip.
- **Versioned skills + governance.** Capsules compound into semver-tagged enterprise skills. Nothing
  auto-publishes on tokens alone — a clearing capsule is staged as a `proposed` version and an **agentic CI**
  A/B harness merges it only if measured reward improves. Multi-developer reconciliation is real: dedup when
  two devs learn the same thing, do/undo conflict resolution by reward + recency, and a reverse **purge**
  pole so the registry never silts up. Every step is ledgered and auditable.
- **The warm-start, made tangible.** An in-app agent chat talks to a local model that already holds the
  latest capsule + recalled tenant memory, with a context inspector that shows *exactly* what the agent sees
  before it generates. That is the observability seam for the whole feature.

| **LLM / ML-engineering depth** | Local-model distillation with chunked map-reduce over long transcripts, an LLM-judge scorer, a measured paired-A/B eval harness, and entity-keyed vector + LLM-guided memory retrieval. |
| **DevOps / cloud / security** | `Dockerfile` + `docker-compose` (app + Ollama) + a handoff `.devcontainer`; an end-of-day CI-gated PR flow; tenant isolation, provenance on every memory, and store-only no-egress writes. |
| **Data structures + systems thinking** | A connected knowledge graph with a single hub invariant, immutable append-only version history, and a reward function (`tokens_saved_downstream − tokens_spent_to_create`) as the only promotion signal. |
| **Communication with stakeholders** | This doc, an honest what's-real-vs-derived breakdown, and a presentation-grade doc set a non-engineer can follow. |

Stack alignment is deliberate: the role lists Python/TypeScript/React/AWS, Docker, and GitHub Actions —
CAPSULE is built natively on the TS/React/Docker/CI half, and the cloud-scale roadmap (hosted Backboard
tenants, the A/B runner moved into GitHub Actions, an AWS-hosted nightly distill) is mapped, not hand-waved.

8090 names its values explicitly — *engineering excellence, insatiable curiosity, bias for action,
ownership, agency, honesty, radical candor.* I built CAPSULE to those, not around them:

- **Bias for action / agency.** I didn't ask whether the gap was worth closing — I designed the layer and

  LLM-judged not trained, the eval is a measured proxy not a p-value claim. I'd rather under-claim and be
  trusted than oversell and be caught.

## Honest scope

The **pipeline is real**: it reads real sessions, distills + scores + stores locally or in live Backboard,
bumps a real local git registry, and opens a real enterprise PR. The labels matter, and I keep them precise:

- **Scoring is LLM-judged, not a trained reward model.**
- **The eval is a multi-sample *measured proxy*** — mean ± stdev over real token counts, consistent-
  direction — not a statistical significance claim.
- **A thin layer is derived** — novelty/importance heuristics, non-A/B reuse estimates, and the
  requirements/work-order scaffolding used to make the demo legible.

[`DATA-REALITY.html`](DATA-REALITY.html) is the canonical, line-by-line breakdown.

## What I'd build first at 8090

CAPSULE is a working argument, not a finished product. If I joined, the first moves I'd push for:

1. **Move the A/B reward harness into GitHub Actions** on the enterprise registry, so skill promotion is
   gated by CI on every PR rather than an end-of-day script.
2. **Hosted, SSO'd Backboard tenants** — per-org assistant isolation so a real team shares one warm memory
   instead of a per-laptop one.
3. **Replace the LLM-judge with a trained reward model** fine-tuned on accepted-vs-rejected capsules — the
   honest next step from "measured proxy" to "learned signal."
4. **An IDE / Factory surface** so warm-start injection happens where engineers already are, not only in the
   in-app chat.
