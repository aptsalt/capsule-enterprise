CAPSULE, every feature explained like you're new here

CAPSULE is the capture + feedback (RL) layer for the 8090 Software Factory: it turns every coding session into a lesson the whole company keeps. One agent solves a problem once; everyone else gets the answer for free, forever. The dataset is real — capsules distilled from the user's actual ~/.claude sessions by a local model, scored by an LLM judge, and promoted into a 28-skill enterprise registry through CI-gated PRs. This page walks each feature in plain words — what it does, why it's needed, where to find it in the app, and what can be improved. Run it locally: cd relay && npm run dev → localhost:3010.

★ The big idea

Every coding session teaches the whole company — a learning loop for a team

Normally, when an AI finishes a coding task, what it learned disappears. The next person starts from zero and pays for the same mistakes again. CAPSULE closes that gap. It runs a loop: do the work, squeeze out the lesson, fold the lesson back into the company's shared skills, and the next task starts smarter. This is the "RL loop" — reinforcement learning, but for a team. The reward it chases is simple and measurable: tokens saved.

1 · Work
An agent does a real coding session — say, making payments safe to retry.
2 · Capture
A local model reads the session and distils a capsule: the one lesson worth keeping.
3 · Improve
The lesson updates a shared skill (a new version) and a technique for people to learn.
4 · Reuse
The next task pulls that skill in and finishes faster, cheaper, on the first try.
↻  The loop compounds.  More sessions → better skills → cheaper sessions → which produce more lessons. In this workspace the savings curve bends from 16,680 tokens in week one (2026-W25) to 43,032 the next (W26).

FEATURE 01

Capsule

A compressed lesson from one session — with two outputs
What it is

A capsule is the squeezed-down takeaway from a single coding session. A long, messy chat transcript goes in; one clear, reusable insight comes out — plus the decisions made, the gotchas hit, and a score for how widely it transfers.

Each capsule produces two outputs: (1) a proposed skill update — "here's how to do this better next time," routed to a specific skill as a version bump; and (2) a technique to learn — the mental model a person or agent should internalise, like "idempotency is an identity problem, not a caching problem."

Think of it as lab notes. The experiment took all afternoon, but the notebook entry is three lines that save the next scientist the whole afternoon.
Why it's needed

Without capsules, hard-won knowledge evaporates when the chat window closes. With them, one painful debugging session — like learning an Angular upgrade needs the local bridge + WebSocket server running — becomes a permanent company asset. In this dataset, capsule CAP-R004 has been reused 5 times, saving ~1,968 tokens each time, and minted the published skill/angular-upgrade@1.0.0.

Where in the app

Left sidebar → "Capsules from today" Each row shows the capsule's ID, the model that authored the work, and a transfer score. Click a real (local) capsule to open its full detail in the Capture panel; click a dataset capsule to focus it in the Knowledge Graph. The blue Capture this session → button at the bottom of the sidebar mints a new one.

What can be improved

When the local model returns thin JSON, the distiller silently backfills fields from a keyword heuristic (cerebras.ts → heuristic()) — useful, but the capsule should flag which fields were inferred so reviewers trust them less. There's also no de-dupe: two near-identical sessions mint two capsules. Next steps: a confidence flag per field, near-duplicate detection, and an inline "edit the finding before storing" step.

FEATURE 02

Local model distillation

Your own machine writes the capsule — private and free
What it is

The job of reading a session and writing the capsule is done by a model running on your own computer — qwen2.5-coder:14b via Ollama. The pipeline tries local first, then an optional cloud boost (Cerebras llama-3.3-70b, only if CEREBRAS_API_KEY is set), then a heuristic last resort so a capsule always renders — even fully offline. Big sessions don't get truncated: a chunked map-reduce reads the whole transcript in pieces and reduces them into one capsule.

Like having an in-house note-taker instead of mailing your private meeting recordings to a stranger to summarise. The notes never leave the building.
Why it's needed

Two big wins. Privacy: a bank's payment code and sensitive data never leave the machine just to be summarised — important when the whole point is keeping card numbers safe. Cost: the summarising step is free, so capturing a lesson never costs more than the lesson saves. Expensive cloud models do the actual building; the cheap local model does the remembering.

Where in the app

Capture panel → "Distilling locally…" When you capture a session, a pulsing state shows Ollama qwen2.5-coder:14b · on-device with the line "No data leaves this machine." Top bar → engine chip A muted mono chip always shows which engine ran the last capture (e.g. Ollama qwen2.5-coder:14b).

What can be improved

A 14B model on a laptop GPU is slow — the distiller allows up to 120 seconds before aborting, and its JSON is lower-quality than the 70B cloud model. Today the spinner is opaque (no token stream, no progress). Next steps: stream partial output, add a quality gate that re-runs on Cerebras when the local handoff score comes back thin, and surface a model picker in the UI instead of an env var.

FEATURE 03

Agentic auto-distill + ambient capture + score gating

Keep the good lessons automatically — drop the noise, no clicks needed
What it is

Not every session is worth remembering. CAPSULE has an agentic mode that decides automatically. Capture is now ambient: a non-blocking Stop-hook in settings.json plus scripts/capture-watcher.ts auto-distills every closed session — no one presses anything. Each capsule is then scored by an LLM judge (scoreCapsuleLLM for transfer, noveltyLLM for novelty; heuristic fallback). The gate is simple: keep the capsule when its transfer score is at or above the threshold, OR when its novelty is ≥ 80. The threshold defaults to 50 and can be set anywhere from 0 to 100.

So a high-transfer lesson (an agent could continue cold) or a genuinely new discovery (novelty 82, 88…) is kept and routed on its own; a thin, low-novelty session is dropped instead of cluttering the registry.

Like a smart spam filter for lessons wired to a doorbell. Sessions get caught the moment they close; obvious keepers and brand-new findings go straight to the shelf; the forgettable ones don't waste anyone's attention.
Why it's needed

If every session minted an adopted skill change, the registry would drown in noise and humans would rubber-stamp everything. Gating keeps the signal high and lets the loop run at scale without a person reviewing every capture — and a clearing capsule is only proposed on a promotion/<skill> ref, with agentic CI publishing it only if the measured reward improves. It's the difference between "a team that learns" and "a team buried in notes."

Where in the app

Capture panel → handoff score + dimensions Every capture shows its overall transfer score and the six dimensions that feed the gate. Right rail → "New capsule ready" A high-novelty capsule like CAP-R004 (novelty 97) surfaces a Review card; the three proposed skills (creative-franchise-expansion, automation-maintenance, api-rate-limiting) route as proposed-not-published versions you can see in the graph's "Routed to" list.

What can be improved

The gate is fully modelled in the store (agenticMode, agenticThreshold default 50, clamped 0–100, with the toggleAgentic toast) and the score/novelty fields exist on every capsule — but there's no visible control yet: no toggle, no threshold slider, no "auto-adopted" audit log. Next step is to surface a slider + a running list of what the gate kept, dropped, and auto-adopted, so the automation is legible and auditable.

FEATURE 04

Knowledge Graph + provenance

A map of how everything connects — and why each version exists
What it is

A live map showing how the pieces link: requirements (what the business asked) → work orders (units of work) → agents (who did it) → skills (reusable know-how) → capsules (the lessons) → memory (where they're stored). Lines show relationships: implements, executes, uses, produces, derives, stores, reads, learns.

Click any skill node and it shows a provenance chain — the answer to "why does this version exist?" — tracing the skill back to the exact capsule, session, finding, work order, and requirement that created it. Nothing is a mystery; every improvement has a paper trail.

Like a family tree for your code's knowledge. Point at any skill and ask "who are your parents?" — and follow the line all the way back to the original business need.
Why it's needed

In a regulated business like a bank, "the AI changed something" isn't good enough — an auditor needs to know why. The graph turns every change into a traceable chain: this skill version exists because that capsule found that insight in that session, serving that requirement. It makes the whole learning loop accountable.

Where in the app

Document header → Knowledge Graph icon The first (network-shaped) icon in the toolbar above the document opens the Knowledge Graph Explorer panel. Click any node for its details and provenance; type filters along the top hide or show node types. Right rail → Actions → Graph / Audit trail shortcuts jump straight to the highest-value skill's trace.

What can be improved

The skill→requirement link is a hard-coded lookup (SKILL_REQ in the panel) rather than being derived from the graph's own links — so adding a skill means editing that map by hand. The chain is also static text; it can't be exported. Next steps: derive provenance from graph.links, highlight the full path on hover, and add a one-click "export audit trail" for regulators.

FEATURE 05

Skills Registry

The company's library of reusable skills
What it is

The shelf of the company's reusable skills — packaged know-how an agent picks up and applies. The enterprise registry holds 28 skills: 13 real (each forged from a real ~/.claude session — Angular Upgrade, Command Verification, API Security, Memory Management, API Rate Limiting, UI Modularity and more) + 15 popular seed skills (REST API design, OAuth2/JWT, Redis caching…). One of the real skills, ui-modularity, was distilled from CAPSULE's own build session. You choose a skill by two things: how many tokens it saves per use, and which use-case it fits. A free-text recommender suggests the right one for whatever you're working on.

Like a cookbook of perfected recipes. You don't reinvent the sauce every dinner — you grab the recipe that already works, plus the notes on what to watch out for.
Why it's needed

Skills are where the learning actually accumulates. A capsule is a single lesson; a skill is the running total of every lesson on a topic. The registry recommends the right skill for the active section, so the agent reaches for proven know-how instead of guessing — that's how one agent's discovery lifts everyone.

Where in the app

Document header → Skills icon The stacked-diamond icon opens the Skills panel: a recommendation banner, an impact note (highest-value skill + Σ saved + adoption), the compounding sparkline, and per-skill cards with token savings, use-case tags, and adopt buttons. Left sidebar → "Search skills by use-case" (Ctrl-K) drives the same recommender.

What can be improved

The recommender is keyword matching (recommendSkill) — it can miss a fit phrased differently and doesn't learn from which skills actually got adopted. There's also no way to author or edit a skill in-app; skills only arrive via captured capsules. Next steps: embeddings-based recall ranked by real usage history, and an in-app skill editor.

FEATURE 06

Enterprise toggle

Flip to the most-learned "capsule-maxxed" best version
What it is

A single switch that says "use the best, most-learned version of every skill." When it's on, the panel serves the capsule-maxxed build — the versions that have absorbed the most lessons and save the most tokens. When it's off, you see the versions pinned in this project. A banner and badge make the active mode obvious (ENTERPRISE · BEST vs PROJECT · PINNED).

Like flipping your team from "good enough" mode to "company-best" mode with one switch — everyone instantly works from the most experienced playbook.
Why it's needed

Learning is worthless if people keep using the old version. The toggle makes adopting improved skills a one-click decision instead of a manual chore — so the savings the loop generates actually reach every seat. It's how a discovery by one agent becomes the default for all 4 seats on the plan.

Where in the app

Skills panel → "ENTERPRISE" switch (top-right) Toggling it re-renders every skill card and the recommendation from the capsule-maxxed versions and fires a toast. The right rail also surfaces one-off "Adopt this newer version?" prompts (e.g. SCA Challenge v1.1.0 — saves 2,600 tokens/use).

What can be improved

The toggle flips a view overlay; it doesn't persist a per-seat or per-team rollout, and there's no staged "canary then everyone" path. Next steps: persist adoption per team, add staged rollout with a rollback, and show how many seats are on each version live.

FEATURE 07

Versions / Version History + compare

Every skill change is numbered and comparable
What it is

Every skill is versioned using semver — three numbers like 2.1.0. Major = a big rethink that changes behaviour. Minor = a real new capability. Patch = a small fix. The newest published version wears a blue Latest badge; a pending one wears Proposed. Tick the Compare box on any two versions to drop into a side-by-side, word-level diff of the changelog and guidance — added words green, removed words struck red.

Like track-changes for know-how. 1.0 → 2.0 rewrites the chapter; 2.0 → 2.1 adds a paragraph; 2.1.0 → 2.1.1 fixes a typo.
Why it's needed

You can see how a skill got smart over time. Command Verification went 1.0.0 (from CAP-R007: inspect system state, don't assume) → 2.0.0 (from CAP-R005: verify the actual current state from authoritative docs, −1,920 tokens/use, published). The bump badge tells you at a glance whether an update is safe to adopt or a bigger behaviour change to review first.

Where in the app

Document header → Versions icon Opens the Version History panel: skill pills across the top, All / Named tabs, day-grouped rows (Today / Yesterday / date), each with its semver chip, bump, adoption count, tokens/use, and a Compare checkbox. + Create Version drafts a named one; Exit Comparison returns to the list.

What can be improved

The diff is set-based (it highlights words present in one side but not the other), so a moved or reordered phrase can mis-highlight — a true line/word LCS diff would be cleaner. + Create Version only fires a toast today, and there's no one-click rollback to a prior version. Next steps: real diff, persistent version creation, and rollback.

FEATURE 08

A/B eval — Capsule vs Cold (multi-sample)

Same task, with the capsule vs without — measured, with error bars
What it is

A head-to-head test in src/lib/eval.ts. The same task is run several times each way: once with the capsule's lesson recalled, once cold (no memory, from scratch). Results come back as mean ± stdev with a consistent-direction check (every sample must point the same way) over real token counts — plus a regression replay of prior capsules.

Like a before-and-after photo taken several times so a lucky single shot can't fool you. Same task, paired runs, error bars — so the benefit isn't a claim, it's measured and repeatable.
Why it's needed

This is the proof the loop works — and these are measured, multi-sample Ollama token counts, not a story. Take AB-03, the API Rate Limiting task:

Where in the app

Document header → A/B Trials icon The bar-chart icon opens "A/B Trials — Capsule vs Cold." Each trial is a card with two columns (with vs cold), pass/fail badges, and a one-line verdict with the token delta. Right rail → Actions → A/B trials opens it too.

What can be improved

The trials carry real, multi-sample measured token deltas (mean ± stdev), but only a few skills have been A/B'd so far — you can't yet A/B a freshly captured capsule on demand from the panel. Next step: wire the panel to the live runner (the eval.ts harness that promote.ts gates on) so any new capsule can be put head-to-head against a cold baseline and the result written back.

FEATURE 09

Backboard memory

The memory follows the team, not the model
What it is

Backboard is the shared memory store where every capsule lives — and these are live writes (real thread_ids) to app.backboard.io/api with an X-API-Key. The key rule: memory belongs to the entity (the tenant/assistant), not to whichever AI model produced it. The enterprise gets one assistant (named capsule); each project is one thread under it (auto-created on first write); skill memories are tagged <skillId>@<semver>. Writes store only the distilled briefing and are storage ops, not inferences (send_to_llm: "false").

Like a company wiki instead of notes in one employee's head. People (and models) come and go; the wiki — and everything the team learned — stays.
Why it's needed

Models change fast. If your knowledge were trapped inside one model, every upgrade would reset you to zero. Backboard makes it model-independent: swap Opus → Sonnet → Haiku and every capsule is still there. Tenant isolation via assistant_id means one tenant never sees another customer's capsules. Backboard's semantic memory is also what dedups near-identical capsules from different developers into one canonical lesson.

Where in the app

Knowledge Graph → "Backboard" memory node The central indigo hub every capsule connects to with a "stores" line and every model with a "reads" line. Capture panel → "stored in Backboard" chip Each captured capsule shows where it landed.

What can be improved

Without a BACKBOARD_API_KEY the store falls back to a local JSON folder (~/.relay/capsules) — fine for the demo, but there's no sync status or retry UI, and recalled memories aren't surfaced back in the composer. Next steps: a connection/sync indicator, and showing which capsule got recalled into the current prompt.

FEATURE 10

Token value

Spent vs saved = the reward signal
What it is

Tokens are what AI work costs. CAPSULE tracks two numbers per capsule: tokens spent to discover the lesson, and tokens saved each time it's reused. Saved-over-time is the scoreboard — and the reward signal the whole loop optimises for (savings = Σ reuses × tokens-saved-per-reuse).

Like "invest once, earn forever." You pay to learn a lesson once; every reuse afterward is dividends. A capsule reused 12 times at 1,800 tokens each has paid for itself many times over.
Why it's needed

It turns "learning" from a vague idea into a number on a dashboard. Across this workspace the loop has saved 43,032 tokens, and the weekly curve keeps bending upward (16,680 → 43,032). That rising line is the compounding effect made visible — the single metric that justifies the whole system, and the A/B token deltas behind it are measured on the local model.

FEATURE 11

Multi-dev registry — local→enterprise promotion, agentic CI, dedup, conflict & purge

A local lesson graduates to the 28-skill enterprise registry — once a day, gated
What it is

Skills live in a repo (addressed like capsule://skills/angular-upgrade) on master — 28 skills (13 real + 15 popular), shared across developers (dee, ven, saim). The enterprise repo holds the capsule-maxxed best versions; each project pins the versions it currently runs. A lesson graduates through a governed, local-first → enterprise pipeline:

  • Local registry first. A kept capsule writes its skill bump to a local registry (~/.capsule/local-registry, branch local-deepak) — instant, private, no network round-trip.
  • End-of-day promote (one PR). scripts/eod-promote.ts batches the day's bumps and opens one CI-gated PR into enterprise master — never a direct push.
  • Agentic CI (test before upgrade). promote.ts gates on the multi-sample A/B eval (new vs current, mean ± stdev) and regression-checks prior capsules, merging the bump only if the measured reward improves; else it stays proposed or is rejected.
  • Dedup & conflict. Near-identical capsules merge into one canonical capsule (ML-001), all authors kept; "do X" vs "undo X / do Y" is a do/undo conflict resolved by reward + recency (ML-002), logged in the MERGE-LEDGER.
  • Purge / retire. scripts/purge-skills.ts + src/lib/purge.ts walk a skill active → deprecated → archived → purged, flagging ABSORBED/SUPERSEDED/UNUSED/LOW_VALUE/ORPHANED into a PURGE-LEDGER (dry-run by default) so the shelf stays clean as it grows.
Like the company's master cookbook with a tasting panel and a librarian: cooks draft in their own notebooks, submit once a day (PR), recipes only enter after a blind taste-test (agentic CI), duplicates get merged into one credited entry — and stale recipes get retired on the record.
Why it's needed

Teams need to move at their own pace without losing the company's best, and a shared registry must not drown in duplicate, contradictory, or stale lessons. Local-first keeps capture instant and private; the once-a-day CI-gated PR means improvements graduate only when measurably better; dedup, conflict resolution and purge keep the registry one clean, authoritative source. Proof it closes the loop: CAPSULE captured its own build session (CAP-SESSION-1a6fcc9b) → skill/ui-modularity@1.0.0, promoted into master via PR #4; dee promoted rest-api-design@1.0.1 + oauth2-jwt-auth@1.0.1.

Where in the app

Skills panel → enterprise/project banner Reads "Enterprise skill set — capsule-maxxed versions" or "This project's active skills — versions pinned in Content Engine," with an ENTERPRISE · BEST / PROJECT · PINNED badge. REPO-FLOW.html · MULTI-DEV.html · REGISTRY-MAP.html render the full master + dee/ven/saim + promotion/* git history, the MERGE-LEDGER and the 28-skill map.

What can be improved

The promotion/CI/dedup/conflict/purge model is real (scripts + ledgers + git history + landed PRs), but is not yet a live in-app surface: there's no MERGE-LEDGER / PURGE-LEDGER viewer, no per-skill promotion/* status panel, and no per-seat rollout. Next step: surface the staging refs, the agentic-CI pass/fail, and both ledgers as a panel so the governance is legible and auditable in the UI.

FEATURE 12

Workspace UX

Doc tabs · read-only mode · @-mentions · resizable panels
What it is

The everyday document-editor chrome that makes CAPSULE feel like a real workspace, not a dashboard:

  • Doc tabs — switch between Requirements, Product Overview, and Technical Requirements across the top.
  • Read-only mode — a document can be editing, read-only (frozen for review), or inactive (archived); the toolbar keys off that status.
  • @-mentions & /-skills — in the composer, type @ to mention a requirement and / to pull in a skill.
  • Resizable / draggable side panels — the side-panel rail is meant to be dragged wider or narrower, clamped between 360 and 680 px so it can never collapse the canvas.
Like the familiar furniture of a doc tool — tabs, a freeze switch, @-mentions, draggable panes — so the powerful stuff sits inside something everyone already knows how to use.
Why it's needed

CAPSULE's value is the loop, but people only adopt tools that feel natural. Tabs keep related docs together; read-only protects a doc under review; @-mentions tie the conversation to real requirements and skills; resizable panels let each person trade document space for graph/skills space. These are the affordances that make the deep features reachable.

Where in the app

Top bar → document tabs · Document header → formatting toolbar + contentEditable surface · Right rail composer → "Use @ to mention requirements, / to use skills". The side panel opens to the left of the right rail and animates in when you click a header icon (Esc closes it).

What can be improved

Several of these are modelled in the store but not yet fully wired in the UI: the doc tabs are local view state and don't swap the document body; read-only/inactive (docStatus) doesn't yet freeze the editor; the @/ affordances show a hint and open the Skills panel but have no autocomplete; and the resizable rail (panelWidth, 360–680) exists in the store while the live grid uses a fixed 360 px column. Next steps: bind tabs to real docs, enforce the read-only freeze, add @/ autocomplete, and wire the drag-resizer to the grid.

FEATURE 13

Agent chat + skills composer

Chat with a local agent that remembers — and load enterprise skills into context
What it is

An in-app agent chat: you talk to a local-Ollama agent that has Backboard memory, so the conversation carries everything the workspace has learned. Alongside it is a skills composer — pick enterprise skills from the 28-skill registry and load them into the chat context so the agent answers with that proven know-how in hand. Replies render as rich Markdown (react-markdown) and whole chats can be saved and reloaded.

It's wired to real routes: /api/chat (the turn), /api/chat/context (assemble memory + loaded skills), /api/chats and /api/chats/save (list + persist). The plumbing lives in src/lib/chatContext.ts, chats.ts and skillCatalog.ts. Shipped in PR #1 (merged).

Like a teammate who has read the whole company wiki — and lets you hand them the exact playbooks for today's problem before they answer.
Why it's needed

Capsules and skills are only valuable if they flow back into the next piece of work. The chat is the recall surface that closes the loop the other way: instead of starting cold, you converse with an agent that already holds the team's Backboard memory and can pull in any enterprise skill on demand. Local Ollama keeps it private and free, so asking is never metered.

Where in the app

Right rail → Chat The agent chat lives in RightPanel. Composer → skills picker loads enterprise skills into the chat context before you send. Chats → save / reload persists a conversation via /api/chats/save.

What can be improved

Today the agent reads Backboard memory and loaded skills, but it doesn't yet write a capsule back from a chat turn, and the skill picker is a flat list rather than the use-case recommender. Next steps: let a useful chat exchange mint a capsule directly, rank the composer's skill suggestions by the active topic, and stream tokens as they arrive.

  2. Follow it through the Knowledge Graph. Open the graph. Click a skill node and read its "Why vX exists" chain — capsule → session → finding → work order → requirement. That's the provenance an auditor wants. Document header → Knowledge Graph icon
  3. Open Skills and flip the Enterprise toggle. Show the 28 registry skills with token savings and the compounding sparkline.   5. Land it on the A/B trial. Open A/B Trials and show AB-03: 247 tokens with the capsule vs 404 cold — 39% cheaper, measured on the local model. Note AB-02 honestly reads 0. End on the 43,032-token total: "this is one company, learning — for real." Document header → A/B Trials icon
