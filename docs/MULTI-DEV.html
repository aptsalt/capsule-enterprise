The multi-developer enterprise model

Many developers learn in their own repos. CAPSULE turns a passing capsule into a proposed skill version, tests it before upgrade with an agentic A/B + regression harness, and reconciles overlapping and contradictory capsules into one canonical registry — auditable down to the commit.

The three questions, answered plainly

1 · How it reaches the enterprise repo

A capsule that clears the gate (transferScore ≥ threshold OR novelty ≥ 80) is proposed, not pushed. It is staged as a PR on a promotion/<skill> ref of master. The head of master only moves when review merges it.

2 · How it's tested before upgrade

Agentic CI runs an A/B harness — new version vs current published — on representative tasks, plus a regression replay of prior capsules. The bump publishes only if measured reward improves (tokens down · pass-rate up · transfer up) with no regressions. CI-for-skills.

3 · Same / contradictory capsules

Dedup: near-identical signatures merge into one canonical capsule; provenance keeps every author. Conflict: a do/undo is flagged and resolved by measured reward + recency — contradictory ⇒ major supersede, ties escalate to a human. Every call is logged in the merge-ledger.

01Promotion — a PR, never a push

Clearing the RL gate produces a PROPOSED skill version staged on a promotion/<skill> ref — the equivalent of a pull request against the enterprise repo. No developer and no agent force-pushes a new version onto master.

1session

Real Claude Code session

A developer's coding session is compressed by local Ollama (qwen2.5-coder:14b) into a capsule — a compressed finding stored in Backboard.

2RL gate

Gate transferScore ≥ threshold · OR · novelty ≥ 80

Only capsules that clear the gate are eligible. Clearing it does not publish — it proposes.

3propose

Staged on promotion/<skill>

Writes PROPOSED-<version>.SKILL.md + a CI-<version>.md report to the staging ref. This is the PR. Master head is untouched.

4agentic CI

A/B harness + regression check

The pipeline measures the proposed version against the current one and replays prior capsules. See section 02.

5review & merge

Human / agentic review merges the bump

Only on reward improved + review OK does master head advance to the new published version.

03Dedup — overlapping capsules from many devs

Every capsule carries a semantic signature. On submit it is matched against the registry and Backboard's semantic memory. Cosine ≥ 0.90 means a re-discovery — the capsules merge into one canonical capsule, provenance keeps all authors, and no redundant version is forged.

Live example · ML-001

ven ⨝ dee on command-verification

  • dee · CAP-R007 — co-originator + pinned 1.0.0 consumer. Finding: "verify by inspecting the actual system state rather than inferred information."
  • ven · CAP-V001 — independent re-discovery of the same principle, submitted 2026-06-27.
  • Both signatures: sig:cmd-verify::inspect-actual-state-over-inference
  • Semantic-memory cosine 0.94 ≥ 0.90 dedup threshold

Resolution

One node, two authors

skill/command-verification
  (canonical: "inspect actual system state")
   ├── dee  — CAP-R007  co-originator, pins 1.0.0
   └── ven  — CAP-V001  deduped 2026-06-27

→ DEDUP: no new version forged
→ Backboard merges 2 facts into 1 node
   with 2 provenance edges
→ stays 1.0.0 for this finding

Forging a second skill would fragment the registry and double-count token savings. Dedup is recorded, not promoted.

04Conflict — do / undo resolution

When capsule A says do X and capsule B says undo X / do Y against the same skill, the pipeline flags a do/undo conflict. It is never a silent minor — it resolves by measured reward + recency, and ties escalate to a human.

Relation → bump policy

major · supersede

Contradictory (do vs undo). Supersede the old version, or reject. Always escalates.

minor

Additive — adds a new, compatible facet to the skill.

patch

Refinement — tightens or clarifies an existing facet.

dedup

Re-discovery — near-identical signature. No version; merge provenance.

Live example · ML-002

saim vs api-rate-limiting

  • DO X · v1.0.0 (CAP-R003) — add client-side backoff + retry on 429s.
  • UNDO X / DO Y · CAP-S001 (saim) — remove client retries; fail fast + honor Retry-After, surface to caller.
  • Gate: novelty 84 ≥ 80, transfer 58 — clears the gate.
  • Relation: do/undo · contradictory

Tie-break & decision

Measured reward + recency

  • Reward: A/B improves on every axis (pass 6→9/10, tokens −35%, transfer +24, no regressions).
  • Recency: CAP-S001 (2026-06-27) is newer than CAP-R003 (2026-06-19). Both favor the new guidance.
  • Decision: SUPERSEDE → 2.0.0 (major), status proposed, staged on promotion/api-rate-limiting — not force-merged.
  • Escalation: contradiction ⇒ human review required to confirm no consumer depends on the old auto-retry contract. On reject, stays 1.0.0 and CAP-S001 is archived rejected with evidence retained.

Every resolution is appended to the append-only MERGE-LEDGER. Invariants: one canonical capsule per finding · no silent contradiction · reward gates publish · entries are never edited in place.
