Software Factory · Context-Handoff Layer

The CAPSULE backend, as it actually runs.

CAPSULE turns a raw AI coding session into a portable, scored, reusable unit of memory. It reads Claude Code transcripts off disk, distills them on a local model, scores how well the understanding transfers, and stores the result as memory that follows the entity — so the next agent starts warm instead of cold.

At a glance

What the backend is responsible for

The whole backend lives in two folders: src/lib/*.ts (the engine) and src/app/api/* (six App Router route handlers that expose it). There is no database server — capsules are written to ~/.relay/capsules on disk and, when a key is present, pushed to Backboard as durable memory. The canonical dataset (the CAPSULE / Content Engine tenant, its 7 skills, 2 agents, A/B trials and metrics) is a typed, immutable module, src/lib/data.ts — distilled from the user's real ~/.claude sessions, not hand-authored.

Request & data flow

The capsule pipeline

One request — POST /api/capsule — drives the entire distillation pipeline. Each stage is a single named export in src/lib. The model call is the only slow step, and it runs on-device.

on-device / memory-critical path route handler / UI surface lib stage
  1. Capture — captureSession(path) reads the .jsonl, counts messages/tools, extracts filesTouched via regex, and compresses to a head+tail transcript capped at 28k chars (where the handoff signal lives).
  2. Distill (local) — distill(raw) sends a strict-JSON system prompt to Ollama qwen2.5-coder:14b with a 120s timeout. Output is parsed into intent / decisions / tried-and-rejected / current-state / next-steps / gotchas / mental-model / open-questions.
  3. Score — scoreCapsule(capsule) rates six transfer dimensions 0–100, picks the two weakest for coaching tips, and renders a verdict.
  4. Store — storeCapsule(capsule) writes to Backboard memory (one thread per project) and always mirrors to local JSON so the demo works offline.
  5. Surface — the route returns {capsule, engine, ms, score, store}; the client prepends it to the Zustand capturedCapsules overlay and opens the Capture panel.

Adoption is a metadata op, not a write

POST /api/skills validates the skill + version and echoes the adopted descriptor (skillId@version tag) — it never mutates data.ts. The client reflects adoption optimistically in the Zustand adopted overlay. This mirrors the memory-follows-the-entity rule: opting an org into a version touches the assistant’s thread metadata, it does not trigger a generation.

The engine

Library modules & responsibilities

capture.tsread & compress

Reads a Claude Code session transcript from ~/.claude/projects/<proj>/<id>.jsonl and extracts the raw material for distillation. Walks each JSONL line, separating USER intents, AI text, and TOOL[name] actions; harvests filesTouched via a path regex (excluding node_modules); computes duration from first/last timestamps; and keeps the head (intent) + tail (latest state) when over 28k chars.

exports · listSessions(limit) · captureSession(path, maxChars) · types RawSession, SessionMeta

cerebras.tsdistill — the local model engine

Despite the filename, the primary engine is local Ollama. distill(session) turns a transcript into a structured HandoffCapsule using a strict-JSON system prompt, then backfills any empty fields from the heuristic so a capsule is never blank. Returns { capsule, engine, ms } where engine is stamped with the model that actually ran.

exports · distill(session) → { capsule, engine, ms }

scorer.ts6-dimension transfer score

A handoff isn’t notes — it’s whether understanding transfers. scoreCapsule(c) rates six cognitive dimensions 0–100 using a richness() heuristic (length + specificity: file refs, :line, numbers, backticks, “because”), averages them into an overall, and emits a verdict plus coaching tips for the two weakest dimensions.

exports · scoreCapsule(c) → HandoffScore · dimensions: intent_clarity, decision_traceability, reasoning_explicitness, gotcha_coverage, next_step_actionability, mental_model_transfer

backboard.tsmemory store + tenant mapping

CAPSULE’s memory component. storeCapsule(c) pushes the DISTILLED briefing into Backboard (POST app.backboard.io/api/threads/messages, X-API-Key, send_to_llm:"false"; thread_id omitted on the first write so Backboard auto-creates + returns a real uuid, then cached) and always mirrors to ~/.relay/capsules/*.json as a fallback. Also creates/caches the single capsule assistant and exports the typed write envelope.

exports · storeCapsule · storeCapsuleMemory · getAssistantId · listLocalCapsules · latestCapsuleForProject · capsuleToBackboardEnvelope · projectThreadKey · skillMemoryTag

capsule.tsthe handoff unit + briefing

Defines HandoffCapsule — the portable, structured record that survives the session boundary — plus the six DIMENSIONS and their labels. capsuleToBriefing(c) renders a compact, agent-ready Markdown briefing (“Decisions already made — do NOT re-litigate”, “Already tried and rejected”, gotchas, mental model, next steps) — this is what gets injected into the next agent.

exports · HandoffCapsule · HandoffScore · DIMENSIONS · capsuleToBriefing(c)

data.tscanonical immutable dataset

The real enterprise world: CAPSULE / Content Engine — requirements, work orders, 8 capsules (CAP-R001…R008) distilled from real sessions, 7 skills with semver version histories (8 versions forged, learnedFrom real capsules), 2 agents, models, MCPs, the knowledge graph, 3 A/B trials with measured Ollama token deltas, and roll-up metrics (tokensSavedTotal 43,032, avgTransfer 46, adoptionRate 63). Never mutated at runtime.

exports · data: Dataset

selectors.tspure derivations over the dataset

Side-effect-free helpers: buildGraph() (canonical nodes/links + deduped capsule→skill learns edges), recommendSkill(query) (blends token savings 40% + transfer 40% + relevance 20%, with section-tag bias), publishedCurrent()/latestVersion() (semver resolution with adoption overlay), provenanceFor() (upstream/downstream lineage), and deterministic fmt/fmtTime formatters for hydration-safe rendering.

The single client coordination contract. Owns all UI/session state — openPanel, panel width, agenticMode + agenticThreshold, the skill adopted overlay, and the capturedCapsules overlay where real captures land (addCapsule prepends, newest first). The canonical data stays immutable; adoption and captures are modelled as overlays on top of it.

Distillation engine

Local-first model, with graceful fallbacks

Distillation runs on the local model by default. The chain tries on-device first, an optional cloud boost second, and a pure-heuristic backfill last — so a capsule always renders, even fully offline with no API keys.

PRIMARY · on-device

Ollama qwen2.5-coder:14b

POST localhost:11434/api/chat, format:"json", temp 0.2, 120s AbortController timeout. Runs with no API key and no network. Engine label: ollama:<model> (local).

OPTIONAL · cloud boost

Cerebras llama-3.3-70b

Attempted only when CEREBRAS_API_KEY is set and the local run returns nothing. Wafer-scale speed; response_format: json_object.

LAST RESORT · no model

Heuristic

Pure string analysis of the transcript — pulls intents, “because/instead” decisions, error/gotcha lines, trailing state. Guarantees a non-blank capsule offline.

// cerebras.ts — distill() resolution order
const r = (await viaOllama(transcript))           // 1 · LOCAL, primary
  || (process.env.CEREBRAS_API_KEY
        ? await viaCerebras(transcript) : null)   // 2 · cloud, only if key set
let d = r?.d || heuristic(session)                  // 3 · heuristic backfill
let engine = r?.engine || "heuristic"
// empty fields are topped up from the heuristic so a capsule is never blank

Environment overrides: OLLAMA_URL, RELAY_OLLAMA_MODEL, CEREBRAS_API_KEY, CEREBRAS_MODEL. The /api/inherit demo uses the same precedence (Cerebras if keyed, else local Ollama) for both the cold and warm agent calls.

Memory model

Backboard — memory that follows the entity

Backboard is CAPSULE’s durable memory substrate. The governing rule across the whole system: memory follows the entity (the assistant / tenant), not the model. Swap the underlying model — Opus → Sonnet → Haiku — and every capsule survives, because context lives in Backboard, not in the weights. That is exactly what makes model selection safe.

Assistant-per-tenant

Every enterprise gets one assistant (here named capsule), created once via POST /assistants and cached in ~/.capsule/backboard.json. The assistant_id is the tenant-isolation boundary — a read/write can only ever touch its own enterprise’s namespace; one tenant never sees another’s capsules. getAssistantId() → cached "capsule" id

Thread-per-project

Each project is one Backboard thread under that assistant. The first write per project omits thread_id so Backboard auto-creates one and returns a real uuid (e.g. d877861b-…), which is cached under a stable local key projectThreadKey(ent, project) = <ent>__<project>. Memory follows the entity (tenant + project), never the model.

send_to_llm = false

A capsule write is a memory op, not an inference op. We persist context for later retrieval; we never trigger a generation on write. It costs storage tokens, not completion tokens.

Skill memories are tagged

Skill memories carry a <skillId>@<semver> tag, pinning a recalled memory to the exact skill version that minted it — which is what makes the registry’s semver bumps auditable end-to-end.

// backboard.ts — the typed write envelope (verified-live shape)
type BackboardEnvelope = {
  assistant_id: "capsule",                   // the single tenant assistant = isolation key
  tags:         ["skill/angular-upgrade@1.0.0"], // pin to skill version
  send_to_llm:  "false",                    // memory write, never inference
  memory:       "Auto",
  content:      "<distilled briefing>"        // only the distilled capsule, never the transcript
  // thread_id omitted on first write → Backboard auto-creates + returns a uuid (cached)
}

Backboard ranks #1 on LoCoMo / LongMemEval — high-recall long-term memory is what lets yesterday’s capsule lift today’s session, which is the token-savings reward the A/B panel surfaces. When no BACKBOARD_API_KEY is present, storeCapsule still mirrors to local JSON so the demo is fully self-contained.

Domain model

The data model

Strict TypeScript in types.ts, instantiated once in data.ts. Every entity is reachable in the knowledge graph; the central hub is Backboard memory.

Capsule

The learnable unit

Each carries a single finding (the novel insight), learnings[], mentalModel, decisions[], gotchas[], and economics: transferScore, novelty, importance, tokensSpent, tokensSavedPerReuse, reuses. The finding is routedTo[] the skills/agents that absorb it, each route proposing a semver bump.

Skill

Versioned, opt-in capability

scope (enterprise/project/org), repoPath (capsule://skills/…), currentVersion, optedIn, adoptionPolicy (auto/manual), and a versions[] semver history. Each version records learnedFrom: {capsule, finding}, changelog, tokenDeltaPerUse, scoreDelta, adoptedBy.

Agent · WorkOrder · Requirement

Who does the work

Requirements (plain-English intent) → Work Orders (units of agent work) → Agents (usesSkills[], executes[], versioned). Each agent version is also derivedFromCapsule — the RL loop reaches agents too.

AbTrial · Metrics · Graph

Proof & topology

AbTrial pairs a withCapsule vs withoutCapsule run (tokens, steps, passed, transferScore). Metrics rolls up tokensSavedTotal, avgTransfer, adoptionRate + weekly compounding. Graph = nodes (requirement/workorder/agent/skill/capsule/model/mcp/memory) + typed links (implements/executes/uses/produces/derives/stores/reads/learns).

Autonomy

The agentic gating loop

Auto-distillation decides whether a freshly-scored capsule is worth keeping and routing without a human in the loop. The gate is a single boolean over two signals already on the capsule: 

// keep / auto-adopt a capsule when EITHER signal clears the bar
const keep = capsule.transferScore >= agenticThreshold   // it transfers well…
          || capsule.novelty >= 80                      // …or it's genuinely new
  • transferScore ≥ threshold — the handoff is good enough to drop into the next agent. Threshold is user-set via the Zustand slider, clamped [0,100] (default 50).
  • novelty ≥ 80 — even a rougher capsule is worth keeping if the finding is genuinely new, so unique knowledge is never discarded on a quality technicality.
Manual vs agentic

With agenticMode OFF, a routed finding waits for human sign-off before a skill version is adopted (RouteStatus: "proposed"). With it ON, findings clearing the gate auto-adopt org-wide — the toast reads “findings ≥ N transfer auto-adopt.” Either way, adoption is recorded in the adopted overlay, never as a dataset mutation.

Compounding loop

Where the skills repo + RL versioning fit

The capsule pipeline is the front half of a reinforcement loop. The back half is the enterprise skills registry: a capsule’s finding is routedTo a skill, proposing a semver bump; once adopted it becomes a new published SkillVersion that future sessions inherit. That is the “RL” — reward (tokens saved, transfer lift) selects which findings get promoted into durable, versioned capability.

  1. Capsule mints a finding. Distillation surfaces one novel, learnable insight per session — e.g. “when upgrading Angular versions, ensure local bridges and WebSocket servers are correctly configured.” (CAP-R004)
  2. Finding routes to a skill. routedTo[] names the target skill and proposes a bump (major/minor/patch) and a proposedVersion, status proposed → adopted.
  3. Promotion, not push. A capsule clearing the gate (transferScore ≥ threshold OR novelty ≥ 80) lands as a proposed version on a promotion/<skill> staging ref of master. Agentic CI then A/B-tests the new version vs current and regression-checks prior capsules, publishing only if the measured reward improves.
  4. Registry publishes a version. A new SkillVersion records learnedFrom: {capsule, finding}, a changelog, and the measured tokenDeltaPerUse + scoreDelta — the reward signal, pinned to skillId@semver.
  5. Agents inherit it. Agents that usesSkills the skill pick up the new version; their own version history is likewise derivedFromCapsule.
  6. A/B proves the lift. The A/B panel runs a task with vs without the capsule on the local model, measuring the real token reduction (e.g. AB-03: 404 → 247 tokens, –39%; AB-01 –16%; AB-02 honestly 0).

Versions are pure semver (compareSemver, latestVersion, publishedCurrent in selectors.ts). The repo address is capsule://skills/<name>; scope can be enterprise, org, or project. The four side panels — Knowledge Graph, Skills, Versions, A/B — are each a read view onto a different slice of this loop.
