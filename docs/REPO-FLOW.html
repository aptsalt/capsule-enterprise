From coding session to enterprise skill — local registry, end-of-day promotion, and agentic CI

Every session compresses into a capsule — distilled and LLM-judged on a local model. A clearing capsule writes its bump to a local registry (local-deepak); the end-of-day promote opens one CI-gated PR on a promotion/<skill> ref of master — a PR, not a push. Agentic CI runs a multi-sample A/B (mean ± stdev) of new vs current and merges the semver bump only if the measured reward improves. The enterprise registry holds 28 skills; personal repos (dee, ven, saim) pull and pin specific versions — and are told when the registry has moved ahead.

The RL loop

1Session

A coding session runs

An agent (e.g. agent/factory-implementer or agent/quality-reviewer) works a work order on claude-opus-4-8.

2Refinery

Capsule Refinery compresses it

The session becomes a capsule (chunked map-reduce over the whole transcript) stored in Backboard — one novel finding plus its mental model, gotchas, and decisions. LLM-judged for transfer + novelty (scoreCapsuleLLM, noveltyLLM).

3Local → EOD PR

Local registry, then one end-of-day PR

Cleared (transferScore ≥ threshold OR novelty ≥ 80) → the bump lands first in the local registry (~/.capsule/local-registry, branch local-deepak). End-of-day, scripts/eod-promote.ts batches the day into one CI-gated PR on a promotion/<skill> ref of master — a PR, not a direct push.

4Agentic CI

Test before upgrade, then merge

Before publishing, promote.ts runs the multi-sample A/B harness (src/lib/eval.ts — new vs current, mean ± stdev, consistent-direction, real token counts) and regression-checks prior capsules. It merges the bump to master only if the measured reward improves; otherwise the version stays proposed or is rejected. Dedup & do/undo conflicts are resolved here (below). One commit per published bump — the audit trail.

5Consume

Personal repos pull & pin

Developers dee, ven, saim each add their my-project/ and pin a few skills in capsule.lock — some intentionally older. capsule status compares the lock against master and reports drift.

Promotion, agentic CI, dedup & conflict — the multi-developer model

promotion · PR not push

Proposed on a staging ref

A capsule clearing the gate (transferScore ≥ threshold OR novelty ≥ 80) is promoted as a proposed version — a PR / commit on a promotion/<skill> ref of master, never a direct push. Human or agentic review merges it.

agentic CI · test before upgrade

CI-for-skills

Before publishing, promote.ts runs the multi-sample A/B harness (eval.ts — new vs current, mean ± stdev, consistent-direction) on representative tasks and regression-checks prior capsules. It merges the bump only if the measured reward improves; else the version stays proposed or is rejected.

dedup · one canonical capsule

Many devs, one lesson

Each capsule carries a semantic signature. Near-identical capsules from dee, ven, saim merge into one canonical capsule with all authors kept as contributors. Backboard’s semantic memory merges the underlying facts — no duplicate knowledge in the registry.

conflict · do / undo

Resolved by reward + recency

When one capsule says "do X" and another "undo X / do Y" on the same skill, the pipeline flags a do/undo conflict and resolves by measured reward + recency (ties → human). Contradictory ⇒ major (supersede), additive ⇒ minor, refinement ⇒ patch. Every resolution lands in a merge-ledger.

Two worked examples from the real data, both in the MERGE-LEDGER. Dedup (ML-001): command-verification was re-discovered by ven (CAP-V001) after dee (CAP-R007) — near-identical signatures merged into one canonical capsule, both authors credited, no new version. Conflict (ML-002): api-rate-limiting got "add client retries" (CAP-R003) then saim's "remove retries, fail fast + honor Retry-After" (CAP-S001) — a do/undo contradiction, resolved by measured reward to a proposed 2.0.0 major, escalated for human review.
