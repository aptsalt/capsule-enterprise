From coding session to enterprise skill — promotion, agentic CI, and how personal repos pin it

Every session compresses into a capsule. An RL gate scores it; a clearing capsule is proposed on a promotion/<skill> staging ref of master — a PR, not a push. Agentic CI A/B-tests the new version vs current and merges the semver bump only if the measured reward improves. Personal repos (developers dee, ven, saim) pull and pin specific versions — and are told when the registry has moved ahead.

The RL loop

1Session

A coding session runs

An agent (e.g. agent/factory-implementer or agent/quality-reviewer) works a work order on claude-opus-4-8.

2Refinery

Capsule Refinery compresses it

The session becomes a capsule stored in Backboard — one novel finding plus its mental model, gotchas, and decisions.

3Gate & propose

The agentic gate proposes a version

Cleared (transferScore ≥ threshold OR novelty ≥ 80) → the finding is routed to a target skill as a proposed bump (major/minor/patch) on a promotion/<skill> staging ref of master — a PR, not a direct push.

4Agentic CI

Test before upgrade, then merge

Before publishing, the pipeline runs the A/B harness (new version vs current) on representative tasks and regression-checks prior capsules. It merges the bump to master only if the measured reward improves (token savings / pass-rate / transfer lift); otherwise the version stays proposed or is rejected. Dedup & do/undo conflicts are resolved here (below). One commit per published bump — the audit trail.

5Consume

Personal repos pull & pin

Developers dee, ven, saim each add their my-project/ and pin a few skills in capsule.lock — some intentionally older. capsule status compares the lock against master and reports drift.

Promotion, agentic CI, dedup & conflict — the multi-developer model

promotion · PR not push

Proposed on a staging ref

A capsule clearing the gate (transferScore ≥ threshold OR novelty ≥ 80) is promoted as a proposed version — a PR / commit on a promotion/<skill> ref of master, never a direct push. Human or agentic review merges it.

agentic CI · test before upgrade

CI-for-skills

Before publishing, the pipeline runs the A/B harness (new vs current) on representative tasks and regression-checks prior capsules. It merges the bump only if the measured reward improves (token savings / pass-rate / transfer lift); else the version stays proposed or is rejected.

dedup · one canonical capsule

Many devs, one lesson

Each capsule carries a semantic signature. Near-identical capsules from dee, ven, saim merge into one canonical capsule with all authors kept as contributors. Backboard’s semantic memory merges the underlying facts — no duplicate knowledge in the registry.

conflict · do / undo

Resolved by reward + recency

When one capsule says "do X" and another "undo X / do Y" on the same skill, the pipeline flags a do/undo conflict and resolves by measured reward + recency (ties → human). Contradictory ⇒ major (supersede), additive ⇒ minor, refinement ⇒ patch. Every resolution lands in a merge-ledger.

Worked example from the real data: command-verification received CAP-R007 ("inspect system state, don’t assume") then CAP-R005 ("verify the actual current state from authoritative docs"). The pipeline read this as a supersede (contradictory guidance), measured the reward, and minted @2.0.0 as a major over @1.0.0 — both kept queryable, the resolution logged in the merge-ledger.
