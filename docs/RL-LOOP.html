Stage by stage

Eleven steps, each a real module or route

No black boxes. Every arrow in the diagram is a named export in src/lib or a handler in src/app/api.

01

Coding SESSION ~/.claude

It starts where the knowledge is born: a real Claude Code run that ends with a head full of decisions, dead-ends and mental model — the context that normally evaporates when the window closes.

02

CAPTURE the transcript

The raw session is read straight off disk from ~/.claude/projects/<proj>/<id>.jsonl. CAPSULE separates USER intents, AI text and TOOL actions, harvests filesTouched via regex, computes duration, and keeps the head (intent) + tail (latest state) capped at 28k chars — where the handoff signal lives.

module · capture.ts — listSessions(limit) · captureSession(path, maxChars)

03

DISTILL on the local model on-device

A strict-JSON system prompt goes to Ollama qwen2.5-coder:14b at localhost:11434 (temp 0.2, 120s timeout, no network, no API key). The transcript becomes a structured HandoffCapsule — intent, decisions, tried-and-rejected, current state, next steps, gotchas, mental model. Empty fields are topped up from a heuristic so a capsule is never blank. Cerebras llama-3.3-70b is an optional cloud boost only when keyed.

module · cerebras.ts — distill(session) → { capsule, engine, ms }

04

SCORE the transfer, 6 dimensions

A handoff isn't notes — it's whether understanding transfers. Each capsule is rated 0–100 across intent clarity, decision traceability, reasoning explicitness, gotcha coverage, next-step actionability and mental-model transfer, using a richness() heuristic (file refs, :line, numbers, "because"). The two weakest dimensions return coaching tips.

module · scorer.ts — scoreCapsule(c) → HandoffScore

05

AGENTIC GATE — keep or skip

Autonomy decides whether a freshly-scored capsule is worth keeping and routing — no human in the loop. A single boolean over two signals already on the capsule:

keep = transferScore ≥ agenticThreshold (slider, default 50)  ||  novelty ≥ 80

Score ≥ threshold means the handoff is drop-in good. Novelty ≥ 80 rescues a rough-but-genuinely-new finding, so unique knowledge is never discarded on a quality technicality.

module · store.ts — agenticMode · agenticThreshold · AGENTIC_MIN/MAX

06

STORE in Backboard memory memory follows the entity

The DISTILLED briefing is pushed to Backboard — the single tenant assistant capsule at app.backboard.io/api (X-API-Key), one thread per project (auto-created on the first write, real uuid returned + cached), send_to_llm:"false" (a memory op, not inference). Swap the model — Opus → Sonnet → Haiku — and every capsule survives, because context lives in memory, not the weights. Always mirrored to ~/.relay/capsules/*.json for offline.

module · backboard.ts — storeCapsule · getAssistantId · projectThreadKey

07

PROMOTION to the enterprise skills repo — PR, not push

A kept capsule's single finding is routedTo[] the skills it should teach — each route naming a target skill and proposing a semver bump (major / minor / patch). Promotion is a proposed version on a promotion/<skill> staging ref of master, not a direct push. Agentic CI then runs the A/B harness (new version vs current) on representative tasks and regression-checks prior capsules, merging the bump only if the measured reward improves — else it stays proposed or is rejected. Near-identical capsules from dee/ven/saim dedup by semantic signature; do/undo clashes resolve by reward + recency in a merge-ledger. The repo address is capsule://skills/<name>.

08

SKILL VERSIONING — semver, learnedFrom

Adoption mints a new published SkillVersion recording learnedFrom: {capsule, finding}, a changelog, and the measured reward — tokenDeltaPerUse + scoreDelta — pinned to skillId@semver. Agents that usesSkills the skill inherit it; their own versions are likewise derivedFromCapsule. Pure semver resolution, fully auditable.

module · selectors.ts (latestVersion, publishedCurrent) · data.ts

09

EXPOSED via the API surface

Six App Router handlers expose every slice of the loop: /api/sessions, /api/capsule, /api/capsules, /api/skills, /api/graph, /api/inherit. Adoption is a metadata op — POST /api/skills echoes the skillId@version descriptor, never mutating the dataset.

routes · src/app/api/*

10

ADOPTED / pinned by personal repos

A developer flips Enterprise on and their personal repo pins the capsule-maxxed best version of each skill. The next session starts warm — the briefing is injected before the first prompt. Adoption lands in the Zustand adopted overlay, optimistically and reversibly.

11

TOKEN SAVINGS = the RL reward, fed back

Reuse of a warm skill measurably cuts tokens — measured on the local model (A/B: 404 → 247 tokens, −39% on AB-03; AB-01 −16%; AB-02 honestly 0). That saving is the reward signal — it's what scoreDelta / tokenDeltaPerUse record, and it's the same harness agentic CI uses to decide which versions publish. Reward closes the loop back onto stage 01: the next session is born smarter.

Why it's reinforcement

The reward is real tokens, not a proxy

Reward selects which findings get promoted — and the promotion makes the next session cheaper still.

Saved tokens close the loop.

Every reuse of a warm skill is measured against a cold baseline. The token reduction is the reward — recorded as tokenDeltaPerUse and scoreDelta on each SkillVersion, compounding 16,680 → 43,032 tokens saved across weeks 2026-W25 → W26 for the real tenant.

reward = tokens(cold) − tokens(warm)
promote if reward > 0 & transfer ≥ thr
agentic CI publishes only if reward improves
next session inherits → cheaper
