1

Capture → distill (local Ollama) → score MEASURED score

CapturePanel.tsx · /api/sessions → /api/capsule · build-real-dataset.ts L181–L191

The panel lists real ~/.claude/projects/*.jsonl sessions, the user picks one, and POST /api/capsule distills it on-device with Ollama qwen2.5-coder:14b — "no data leaves this machine." Oversized sessions self-route through a chunked map-reduce distiller (distillChunked) so a whole long session is captured, not truncated. The distilled capsule is then scored by an LLM-judge (scoreCapsuleLLM — a local model rating the handoff, not a trained reward) across six cognitive dimensions, with a heuristic fallback when Ollama is down; the overall is the transferScore. Capture also runs ambient: a non-blocking Stop-hook + capture-watcher.ts auto-distill closed sessions with no panel open.

Server-side, the dataset builder runs the same contract headless — capturing the transcript, distilling, then scoring:

It then asks the local model for the single most important reusable finding — one concrete sentence (build-real-dataset.ts L196–L202) — which becomes the unit that everything downstream routes, versions, and A/B-tests.

2

The agentic gate KEEP / SKIP

CapturePanel.tsx L146 (bar) · L237–L249 (decision)

This is the decision point. In agentic mode every session is auto-distilled, then gated. A capsule is kept (promoted to the enterprise repo) only if it clears the transfer threshold or carries a uniquely strong proposition — novelty ≥ 80. Skipped capsules are dropped, never added to the overlay.

Novelty, honestly. The /api/capsule response has no first-class "novelty" field, so noveltyOf() (L156–L160) derives it as the capsule's strongest single dimension — its most distinctive signal. A capsule can be ordinary on average yet exceptional on one axis (a rare gotcha, a crisp mental model); that single spike is what lets it clear the gate below the transfer bar. Demo heuristic over local output, not a model-reported metric.
3

Route finding → skill + semver classification

build-real-dataset.ts L267–L284 (classify) · L130 (applyBump) · L280 (new = major)

Each kept finding is classified by the local model against the existing skill library. If it clearly belongs to a skill it reuses that exact skillId and picks minor/patch; otherwise it invents a new kebab skillId and uses major.

Version timelines are built per skill, sorted by real capsule createdAt (L290–L312). A version whose source capsule scored transferScore < 45 is staged as proposed rather than published (L298) — the registry never auto-trusts a thin handoff.

4

Store to live Backboard MEASURED thread_ids

build-real-dataset.ts L446–L470 · capsuleMemoryBriefing / storeCapsuleMemory

Only the distilled briefing — never raw transcripts — is persisted to the real Backboard memory hub, and stored with send_to_llm:false so it lands as durable tenant memory rather than being replayed into a model context. Real thread_ids come back and are pinned onto the capsule; failures degrade gracefully to a local thread.

Why send_to_llm:false. Memory follows the tenant, not the model. The briefing is a durable, queryable artifact in Backboard (Memory Pro) — it powers dedup's semantic-cosine matching in Stage 6, but it is not auto-injected into prompts.
5

Promotion as a PR staging ref

PROMOTION.md · promotion/<skill>/PROPOSED-x.y.z + CI-x.y.z

Clearing the gate does not push to master head. It proposes a version on a promotion/<skill> staging ref — the equivalent of a PR. No developer and no agent force-pushes a new version. The head only moves when the proposed version passes agentic CI and review merges it.

What gets staged

promotion/<skill>/PROPOSED-<version>.SKILL.md — the candidate skill body

promotion/<skill>/CI-<version>.md — the agentic-CI report

Agentic CI — test before upgrade

  • Multi-sample A/B (src/lib/eval.ts): N paired runs, new vs current, on the local model — reports mean ± stdev + direction-consistency across all N (honest, not a p-value), real prompt_eval+eval token counts
  • Regression check: LLM-judge replay of prior capsules the skill serves — must not break learned behavior
Merge rule (CI-for-skills). A version is published only if measured reward improves (tokens down / pass-rate up / transfer up) and no regressions. Otherwise it stays proposed or is rejected. A version upgrade must earn its place with evidence — exactly like code passing tests.
On-demand LIVE promotion. /api/promote makes stages 5–6 execute for real on demand — a genuine git commit + push that writes the promotion/<skill>/ staging artifacts (and, on a passing merge, advances master head). The same governance applies: it stages a PR, it does not bypass the gate or the ledger.
LOCAL → END-OF-DAY → ENTERPRISE. During the day every earned upgrade is committed to the developer's local registry (~/.capsule/local-registry, branch local-deepak) via bumpSkillLocal — real local commit, not pushed. The EOD job (scripts/eod-promote.ts) pushes the branch and opens one CI-gated PR local-deepak → master against the enterprise registry; merging it publishes the day's learning. A purge/retire pole (active → deprecated → archived → purged, logged in a PURGE-LEDGER) keeps the 28-skill registry from growing dead weight — Backboard provenance is never touched.

Ledger invariants

The four rules the MERGE-LEDGER guarantees, no matter how many developers contribute.

  1. One canonical capsule per finding. Dedup collapses re-discoveries; provenance keeps every author.
  2. No silent contradiction. A do/undo can never be a minor/patch — it is supersede or reject, and it escalates.
  3. Reward gates publish. A proposed version merges to master head only if the A/B harness shows improved reward and prior-capsule regressions pass.
  4. Append-only. Entries are never edited in place; corrections are new entries (MERGE-LEDGER + PURGE-LEDGER).
  5. It runs on itself. CAPSULE captured its own build session (CAP-SESSION-1a6fcc9b) → skill/ui-modularity@1.0.0, promoted into enterprise master via PR #4 — the loop, proven end to end on its own code.
